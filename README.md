## Описание решения

### Подход

Решение задачи как гибридный ансамбль признаков и моделей:

1. Из названия номера `supplier_room_name` строятся ручные признаки (уровень номера, тип ванной, кровати, вместимость, вид, балкон, семейность и др.).
2. Добавляется контекст отеля: признаки, показывающие, насколько описание комнаты отличается от среднего профиля по конкретному `hotel_id`.
3. Добавляется компактный трансформер (`cointegrated/rubert-tiny2`) и используется как генератор сильных текстовых признаков (OOF-вероятность, логит, margin, entropy).
4. Считаются фикс. эмбеддинги текста и далее сжимаются через PCA до 32 компонент.
5. Все признаки объединяются для получения финального `CatBoost`
6. Отдельно подбирается рабочий порог по правилу `precision >= 0.95` с максимальным recall.

### Архитектура решения

Архитектура состоит из четырех компонентов:

1. **Feature Engineering слой**

- Нормализация текста и извлечение доменных атрибутов комнаты.
- Формирование hotel-aware признаков (сравнение с профилем отеля).

2. **Text Encoder слой**

- KFold-обучение трансформера для снижения рисков утечки и завышения метрик (OOF).
- Генерация трансформерных мета-признаков на train/test.

3. **Embedding слой**

- Извлечение плотных эмбеддингов тем же encoder.
- PCA-сжатие и добавление компонент в табличные фичи.

4. **Final Classifier слой**

- Обучение CatBoost на гибридном наборе фич.
- Валидация по PR-AUC, подбор порога, переобучение модели на полном train.

Пайплайн обработки данных:

1. Читаем `public_dataset.csv` и test-файл.
2. Строим базовые табличные признаки из текста.
3. Получаем OOF-прогнозы трансформера и добавляем их как признаки.
4. Получаем PCA-эмбеддинги и добавляем их как признаки.
5. Сохраняем единые train/test hybrid-таблицы признаков.
6. Обучаем CatBoost, считаем метрики, сохраняем feature importance.

### Ключевые особенности

- Гибридная модель: ручные признаки \+ трансформерные сигналы \+ эмбеддинги в одном CatBoost.
- OOF-генерация текстовых признаков, чтобы избежать leakage и завышенных метрик.
- Оптимизация именно под бизнес-метрику `PRAUC`, а не только под ROC-AUC.
- Настройка рабочего порога под ограничение `precision >= 0.95`.
- Учет дисбаланса классов через `auto_class_weights=Balanced`.
- Отдельная страховочная ветка `no_hotel_context` с grouped CV по `hotel_id` для проверки устойчивости на новых отелях.

### Итоговые метрики на random 20% holdout публичных данных

Для основной гибридной модели на валидации по схеме **80/20** получены следующие результаты:

* **PR-AUC:** `0.96723`
* **Recall при Precision \>= 0.95:** `0.8140`
* **Рабочий порог:** `0.9007`
